# Documentation Review Prompt

Use this prompt with another LLM to analyze the MMBR web-platform documentation for improvements and gaps.

---

## Prompt

You are a senior software architect reviewing the documentation of a Next.js web application called MMBR (MemBrain). The project was initialized with the Qubika Agentic Framework (QAF), which auto-generated AI development artifacts (CLAUDE.md, skills, agents, commands) from codebase analysis.

Your task is to review the documentation suite for completeness, accuracy, consistency, and gaps. The project uses Claude Code as the primary AI development tool, so documentation quality directly impacts AI-assisted development productivity.

### Project Context

- **Product:** AI-powered RAG chat for wastewater treatment plant operators
- **Stack:** Next.js 16 (App Router), TypeScript, Tailwind CSS, Auth0, PostgreSQL, Drizzle ORM
- **External dependency:** QBricks RAG Agent (owns conversations, messages, citations)
- **AI tooling:** Claude Code with QAF-generated skills, agents, and commands
- **Team:** 1 frontend dev, 2 backend devs, 1 data engineer, 1 SRE, 1 product/AI lead

### Documentation Structure

The documentation lives in two locations within the `web-platform/` repository:

**`.claude/` — AI development artifacts (auto-generated by QAF, then refined)**
1. `CLAUDE.md` (100 lines) — Quick reference: tech stack, file placement guide, directory structure, commands, Source of Truth index
2. `project-context/SKILL.md` (373 lines) — Deep architectural knowledge: request lifecycle, auth patterns, QBricks proxy, 6 gotchas, workflow templates, testing strategy, conventions
3. `design-system.md` (238 lines) — Figma design tokens: colors, typography, spacing, shadows, component-specific styles
4. `agents/` (5 files) — QAF agents: planner, 3 implementers (TypeScript, JavaScript, generic), visual-verifier
5. `commands/` (2 files) — `implement-ticket.md`, `create-sdd-ticket.md`
6. `skills/` (7 categories, 20+ skills) — Foundation, development workflow, quality assurance, language/frameworks, documentation, infrastructure, Next.js best practices
7. `framework-config.json` — QAF configuration with stack profile and agent model mapping

**`.kiro/specs/` — Requirements and implementation plan**
8. `requirements.md` (429 lines) — 25 requirements with acceptance criteria across 4 modules (Auth, Chat, Documents, Profile)
9. `tasks.md` (331 lines) — 42 implementation tasks with subtasks, requirement traceability, and progress checkboxes (30 done, 12 remaining)
10. `design.md` (1104 lines) — Full design document: SQL schema with ERD, component type signatures, 29 correctness properties mapped to requirements, API contracts, error handling scenarios

**`docs/` — Architecture**
11. `ARCHITECTURE.md` (400 lines) — System architecture: data ownership split (Auth0 vs PostgreSQL vs QBricks), BFF pattern, integration diagrams

### Review Criteria

For each document, evaluate:

1. **Completeness** — Does it cover everything it should for its purpose? Are there missing sections?
2. **Accuracy** — Are there statements that might be outdated or incorrect given the stack described?
3. **Consistency** — Do the documents contradict each other? Are naming conventions, patterns, and decisions aligned across all files?
4. **Gaps** — What knowledge is missing entirely? What would a new developer (or AI agent) struggle with?
5. **Redundancy** — Is there unnecessary duplication that could drift out of sync?
6. **QAF alignment** — Are the auto-generated artifacts (agents, skills, commands) well-suited for this specific project? Are there QAF capabilities not being leveraged?

### Specific Questions to Answer

1. Does the CLAUDE.md Source of Truth index accurately reflect the most important documents? Is anything missing?
2. Is there a clear path from "I have a Jira ticket" to "I have a merged PR" documented across these files?
3. Are the 29 correctness properties in design.md still aligned with the 25 requirements in requirements.md?
4. Does SKILL.md accurately describe the codebase patterns, or are there gotchas missing?
5. Is the QBricks integration documented well enough for someone to connect the frontend to real APIs?
6. Are there conventions or decisions made but not documented anywhere?
7. What would break if a developer followed only these docs without reading the code?

### Output Format

Provide your review as:

1. **Executive Summary** (3-5 sentences)
2. **Findings Table** — One row per finding with: Severity (Critical/High/Medium/Low), Document, Finding, Recommendation
3. **Gap Analysis** — What's entirely missing
4. **Redundancy Report** — What overlaps and risks drift
5. **Top 5 Actionable Recommendations** — Prioritized by impact
