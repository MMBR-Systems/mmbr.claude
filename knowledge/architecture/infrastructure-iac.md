# Infrastructure IaC — `MMBR-Systems/infraestructure-iac`

## What it is

The terraform repo that defines the AWS infrastructure for the MMBR project. As of 2026-05-07 it manages **most** of the AWS-side resources for `web-platform`, `qbrick`, and `backend-ui` across `dev`, `qa`, and `prod`. Source: <https://github.com/MMBR-Systems/infraestructure-iac>.

## What's IaC-managed today

After PR #2 was merged on 2026-05-07, the repo manages:

- **ECS:** cluster, the three services (`web_platform`, `qbrick`, `backend_ui`), task definitions, autoscaling targets and policies (CPU + memory per service), CloudWatch log groups
- **Secrets Manager:** `mmbr-<env>-web-platform`, `mmbr-<env>-qbrick`, `mmbr-<env>-backend-ui`, `mmbr-<env>-valkey-auth-token`
- **Compute:** Aurora cluster + RDS Proxy, ElastiCache Valkey, EC2 bastion
- **Network:** VPC, subnets, security groups, ALB + listener rules + target groups, ACM certificates, Route53 records
- **IAM:** task roles per service, execution role, GHA OIDC role (in `common/`)
- **Observability:** CloudWatch metric alarms (ALB 5xx, response time, Aurora connections / CPU, ECS CPU + memory)
- **Service discovery:** internal namespace + service registrations for service-to-service traffic

## What's deliberately outside or partial

- **Image tags** are not in terraform files. They live in SSM parameters (`web_platform_image_tag`, etc.) which the deploy CI pipelines write. Terraform reads them via `data.aws_ssm_parameter`. This decouples app deploys from infra changes.
- **Secret values** (`secret_string`) are written by terraform on first creation but then `lifecycle { ignore_changes = [secret_string] }` keeps subsequent applies from overwriting them. Manual edits to values via the AWS console are intentionally allowed; only the secret container (name, recovery window) is reconciled.
- **Per-workflow QAP credentials** in `qbrick`'s Postgres (the `QAP_API_KEY`/`QAP_API_SECRET`/`QAP_AGENT_ID` bundle) are generated by qbrick at runtime via `POST /workflow_descriptions/{id}/generate_api_key`. Out of terraform's scope.
- **Per-env Auth0 application config** lives in the Auth0 dashboard (callback URLs, allowed origins, etc.), not in terraform. See `reference/setup/auth0-dashboard.md` for the manual procedure.
- **AWS console–only edits** still happen during incident response (today, 2026-05-07, several manual edits to `dev`/`qa`/`prod` secrets). Terraform's `ignore_changes = [secret_string]` is what keeps these from being clobbered. The risk of drift in non-`secret_string` attributes remains.

## How changes flow today

1. Engineer opens a PR in `infraestructure-iac` with a `.tf` change.
2. CI (`terraform-ci.yml`) runs `terraform fmt`, `terraform validate`, `tflint`, `trivy`. **No plan, no apply.**
3. After approval, PR is merged to `main`.
4. **Terraform owner (currently Leandro)** pulls `main` locally and runs:
   ```bash
   cd env/<env>
   terraform init -backend-config=backend.tfvars
   terraform plan  -var-file=common.tfvars -var-file=terraform.tfvars
   terraform apply -var-file=common.tfvars -var-file=terraform.tfvars
   ```
   Once per env (dev, qa, prod). Manually.
5. State is stored in S3 buckets (`mmbr-terraform-state-<env>`), versioned, encrypted.

This is a deliberate decision — see `decisions/why-no-terraform-ci-pipeline.md` for the rationale and the four triggers that should reopen it.

## Naming convention

Almost everything is parameterized by `${var.app_prefix}-${var.env_name}-<resource>`:

| Resource | Example (dev) |
|---|---|
| Secret | `mmbr-dev-web-platform` |
| ECS cluster | `ecs-dev` (cluster name omits `mmbr` prefix) |
| ECS service | `web-platform-dev` |
| Task family | `dev-web-platform` |
| Log group | `web-platform-dev` |
| RDS Proxy host | `mmbr-dev-rds-proxy.proxy-cfq0o6q0ocfv.us-east-2.rds.amazonaws.com` |
| Valkey | `mmbr-valkey-dev` |
| ALB | `ecs-dev-alb` |

Resource names in terraform code use snake_case (`aws_ecs_service.web_platform`); the resulting AWS resource gets the rendered string per the convention above.

## Where state and secrets live

- **State buckets:** `mmbr-terraform-state-{dev,qa,prod}` in `us-east-2`, encrypted with the AWS-managed S3 KMS key. Versioning enabled. Configured in each env's `backend.tfvars`.
- **Secret values:** AWS Secrets Manager. Three categories:
  1. App-level (`mmbr-<env>-{web-platform,qbrick,backend-ui}`) — written first by terraform, then maintained manually because of `ignore_changes`.
  2. RDS-managed master credential (`rds!cluster-<id>`) — auto-rotated by RDS, terraform reads it via `data.aws_secretsmanager_secret_version.aurora_master`.
  3. Valkey auth token (`mmbr-<env>-valkey-auth-token`) — generated as a `random_password` in terraform.

## Common operations

- **"What will change if I apply?"** Open a PR, then run `terraform plan` locally on the branch. CI does not show plan output (decision intentional).
- **"Where is X defined?"** `grep -rn "X" globals/ env/`. Almost everything lives in `globals/`; only env-specific values and per-env shared module configuration is in `env/<env>/`.
- **"How do I add a new env?"** Create `env/<new>/` with the four `.tfvars` files, mirror the values you want from `env/qa/`, run `terraform apply` there. Resources from `globals/` get instantiated for the new env.
- **"How do I rotate a secret value?"** Edit the value in the AWS console (or via `put-secret-value`). Terraform's `ignore_changes = [secret_string]` won't clobber it on the next apply. ECS service redeploy needed for containers to pick up the new value.
- **"How do I add a new app/service?"** Add the four blocks to `globals/`: `aws_secretsmanager_secret`, `aws_ecs_task_definition`, `aws_ecs_service`, autoscaling block. Add CloudWatch log group and alarms. Update outputs. PR + merge + apply.
